# Centrifugo PRO v6.8.0 — Release notes

This release brings Centrifugo PRO enhancements on top of the OSS [v6.8.0](https://github.com/centrifugal/centrifugo/releases/tag/v6.8.0) foundation: production-grade scaling for the new map and shared poll subscriptions, server-side publication filtering for per-subscriber RBAC, and a notification fast path that drops shared poll latency from seconds to milliseconds.

See the [OSS v6.8.0 release notes](https://github.com/centrifugal/centrifugo/releases/tag/v6.8.0) for the new subscription primitives (map, shared poll) and PostgreSQL components (stream broker, map broker, controller) — all of those ship in OSS and are available unchanged in PRO.

## Highlights

- **[In-memory cache layer for map subscriptions](https://centrifugal.dev/docs/pro/map_subscriptions#in-memory-cache-layer)** — map state served from memory on each node, microsecond-range read latencies.
- **[Full-state delta sync](https://centrifugal.dev/docs/pro/map_subscriptions#full-state-delta-sync)** — derived stream channel that delivers the entire map collection as a Fossil delta per tick. Massive bandwidth savings for cursors, game state, fleet dashboards.
- **[Server-side publication tags filter](https://centrifugal.dev/docs/pro/server_tags_filter)** — per-subscriber RBAC for stream and map subscriptions. Backend sets the filter, client cannot override.
- **[Shared poll notification fast path](https://centrifugal.dev/docs/pro/shared_poll#notification-fast-path)** — application-pushed signals trigger immediate backend polls for affected keys. Drops update latency from seconds to milliseconds.
- **[Shared poll relay](https://centrifugal.dev/docs/pro/shared_poll#shared-poll-relay)** — standalone process that centralizes backend polling for the whole cluster. Backend is called once per cycle, not once per node.
- **[PostgreSQL broker fan-out & read replicas](https://centrifugal.dev/docs/pro/map_subscriptions#postgresql-enhancements)** — scale PostgreSQL-backed brokers across larger clusters and replicas.
- **[PostgreSQL controller read replicas](https://centrifugal.dev/docs/pro/scalability#read-replica-support)** — outbox polling on a PostgreSQL replica while writes stay on the primary.
- **[Per-namespace map brokers](https://centrifugal.dev/docs/pro/map_subscriptions#per-namespace-map-brokers)** — different namespaces can use different map broker backends.

## Map subscriptions PRO

### In-memory cache layer

Channel state is kept in memory on each Centrifugo node, populated from the configured map broker. Read latencies drop from milliseconds to microseconds. Trade-off is proportional memory usage on each node.

```json title="config.json"
{
  "map_broker": {
    "type": "postgres",
    "postgres": { "dsn": "..." },
    "cache": {
      "enabled": true,
      "max_channels": 1000,
      "idle_timeout": "10m",
      "sync_interval": "30s"
    }
  }
}
```

The cache is filled via PUB/SUB and periodically reconciled with the backend as a safety net. Stream offsets in the cache match write order in the main storage, keeping incremental recovery correct.

**Docs:** [Map subscriptions PRO — In-memory cache layer](https://centrifugal.dev/docs/pro/map_subscriptions#in-memory-cache-layer)

### Full-state delta sync

A derived stream channel that delivers the entire map collection as a single delta-compressed publication per tick. Instead of receiving individual per-key updates, subscribers get one compact Fossil delta per tick — the minimum bytes needed to go from the previous snapshot to the current one.

Configured via `full_state_channel_prefix` on the map namespace. The derived channel speaks the standard stream subscription protocol — no special SDK API.

```javascript
const sub = client.newSubscription('fullstate:cursors:room1', {
  delta: 'fossil',
});
sub.on('publication', (ctx) => renderAllCursors(ctx.data));
```

Best fit for collections where clients render the full state every frame — cursors, game state, IoT fleet dashboards, scoreboards. Bandwidth savings come from eliminating per-message protocol overhead and built-in dedup (only the latest value per key per tick).

**Docs:** [Full-state delta sync](https://centrifugal.dev/docs/pro/map_subscriptions#full-state-delta-sync)

### Per-namespace map brokers

Different map namespaces can use different broker backends — for example, ephemeral cursor data in Redis and persistent scoreboard state in PostgreSQL.

**Docs:** [Per-namespace map brokers](https://centrifugal.dev/docs/pro/map_subscriptions#per-namespace-map-brokers)

### PostgreSQL broker enhancements

Apply to both the map broker (`map_broker.postgres`) and the [stream broker](https://centrifugal.dev/docs/server/engines#postgresql-broker) (`broker.postgres`):

- **[Read replicas](https://centrifugal.dev/docs/pro/map_subscriptions#read-replicas)** — read load distributed across PostgreSQL replicas via consistent hashing on channel name.
- **[Broker fan-out](https://centrifugal.dev/docs/pro/map_subscriptions#broker-fan-out)** — only one node per shard polls PostgreSQL, others receive updates via Redis or Nats. PostgreSQL polling load scales sub-linearly with cluster size.

### Redis enhancements

- **Redis Cluster support** for the Redis map broker via sharded PUB/SUB (the OSS Redis map broker supports only a single Redis instance or client-side consistent sharding).
- **[Subscribe on replica](https://centrifugal.dev/docs/pro/scalability#subscribe-on-replica)** for the Redis map broker — PUB/SUB subscriptions move to replica nodes.
- **[Node-grouped sharded PUB/SUB](https://centrifugal.dev/docs/pro/scalability#node-grouped-sharded-pubsub)** — also applies to the Redis map broker.

## Shared poll subscriptions PRO

### Cached latest data

`keep_latest_data` enables Centrifugo to cache the latest data and version per tracked item in memory:

- **Instant data for new clients** — when a client tracks keys with version `0`, cached data is returned in the track response. No backend call, no waiting for the next poll cycle. (Requires `versioned` mode.)
- **Delta compression** — Fossil deltas computed between the previous and current cached versions. Clients that negotiated delta compression receive compact patches instead of full payloads.

Particularly valuable for config sync (single key, long refresh interval, instant on connect), reconnect/page navigation, and low-frequency polling channels.

**Docs:** [Cached latest data](https://centrifugal.dev/docs/pro/shared_poll#cached-latest-data)

### Notification fast path

Application-pushed signals trigger immediate backend polls for affected keys, dropping update latency from seconds to milliseconds without abandoning the polling model.

Notifications are **not data** — they are channel and key hints. Your application publishes a small JSON message to a Redis or NATS pub/sub channel; Centrifugo subscribes, batches incoming notifications, and triggers an immediate backend poll for the notified keys.

```bash
redis-cli PUBLISH shared_poll_notify '{"items":[{"channel":"post_votes:feed1","key":"post_123"}]}'
```

Per-namespace batching options (`batch_max_size`, `batch_max_delay`) let you trade latency for backend efficiency. Timer-based polling continues running in parallel — notifications are an acceleration layer, not a replacement.

**Docs:** [Notification fast path](https://centrifugal.dev/docs/pro/shared_poll#notification-fast-path)

### Shared poll relay

A standalone Centrifugo process that centralizes backend polling. Backend is called once per refresh cycle, not once per node. Run with `centrifugo --config config.json --mode=shared_poll_relay`. Normal nodes redirect refresh requests to the relay automatically when `shared_poll_relay.enabled` is `true`.

The relay also maintains version history per item, providing `prev_data` for delta compression — no backend changes needed.

**Docs:** [Shared poll relay](https://centrifugal.dev/docs/pro/shared_poll#shared-poll-relay)

### Adaptive backpressure

When a refresh cycle takes longer than the configured `refresh_interval`, the interval automatically extends to prevent overloading the backend. Utilization-based: under 50% recovers toward the configured value, 50–100% stretches proportionally, over 100% doubles up to the cap. When the backend recovers, the interval shrinks back.

**Docs:** [Adaptive backpressure](https://centrifugal.dev/docs/pro/shared_poll#adaptive-backpressure)

## Server-side publication tags filter

Per-subscriber publication filtering — set by your backend via subscribe proxy, JWT, or connection token, and enforced by Centrifugo. Unlike the [client-side tags filter](https://centrifugal.dev/docs/server/publication_filtering) (a bandwidth optimization the client controls), the server-side filter cannot be overridden by the client. It's the access-control mechanism for per-subscriber RBAC within shared channels.

Works for **both stream and map subscriptions**.

```json title="subscribe proxy response"
{
  "result": {
    "server_tags_filter": {
      "op": "and",
      "nodes": [
        {"key": "team",  "cmp": "eq",  "val": "engineering"},
        {"key": "level", "cmp": "gte", "val": "5"}
      ]
    }
  }
}
```

For map subscriptions, the filter applies across all three sync phases — state, stream catch-up, and live. Removal events carry the original entry's tags, so subscribers only receive removals for keys they were authorized to see.

**Docs:** [Server-side publication tags filter](https://centrifugal.dev/docs/pro/server_tags_filter)

## PostgreSQL controller — read replicas

The OSS [PostgreSQL controller](https://centrifugal.dev/docs/server/engines) gets a PRO addition: outbox polling can be routed to a PostgreSQL replica while writes stay on the primary.

LISTEN/NOTIFY always uses the primary (PostgreSQL limitation), but the polling query runs on the replica.

**Docs:** [PostgreSQL controller — read replica support](https://centrifugal.dev/docs/pro/scalability#read-replica-support)

## Other improvements

- **`client_publish_debounce_interval`** namespace option — client-side debouncing for client publications. When clients publish to a map channel rapidly (cursors, typing indicators), this option may help to reduce the traffic from server side.

## SDK support

Same as OSS: map subscriptions and shared poll subscriptions are currently supported on the client side only by `centrifuge-js`.

## Migration notes

This release is backward-compatible. No required migrations. All new features are opt-in.
